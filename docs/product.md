# What The Product Does

[Docs index](README.md) | [Quickstart](quickstart.md) | [How it works](how-it-works.md) | [Runtime policy](runtime-policy.md)

Guardrail is a service for giving an agent usable onchain wallet access without giving that agent a fully privileged hot key.

In the current product shape, Guardrail does three things together:

1. creates a smart-wallet setup for one agent workflow,
2. keeps durable ownership with a human through a passkey,
3. gates the agent runtime path behind backend policy checks and backend co-signing.

## The Concrete Service We Provide

Guardrail is not just a library and not just a frontend.

The product is the full flow that combines:

- a CLI that an agent or operator uses to create and use wallets,
- a backend that provisions wallet requests, enforces runtime policy, and co-signs allowed actions,
- a provisioning frontend where the human creates the owner passkey,
- a public `skill.md` endpoint on the main frontend for agent-readable onboarding instructions,
- a smart-account configuration that splits owner control from runtime agent access.

The service being sold by the product is:

- "give this agent a wallet it can use,"
- "keep the human as the durable owner,"
- "do not let the agent act outside the policy we configured."

## Who Does What

### Human

The human is the durable owner.

Today that means:

- the human opens the provisioning link,
- the human creates a passkey,
- that passkey becomes the owner path for the wallet.

The human is not required for each runtime action after provisioning, but the human remains the admin path.

### Agent

The agent gets a local key generated by the CLI and stored in the local wallet state.

That key is used to request runtime actions such as:

- contract calls that fit the configured allowlist,
- specific supported official USDC actions,
- x402 payment flows built on the supported USDC path.

The agent key alone is intentionally not enough to use the runtime path.

### Guardrail Backend

The backend is an active part of the trust model.

Today it:

- creates a backend co-signer per wallet request,
- stores wallet requests and provisioning state,
- checks whether a requested runtime action fits policy,
- records policy consumption for budgeted actions,
- co-signs only the actions it allows,
- exposes restricted RPC and bundler relay endpoints for supported chains.

## Lifecycle Of A Wallet

The current wallet lifecycle is:

1. The CLI creates a wallet request with a policy.
2. The backend creates the matching wallet configuration and returns a provisioning link.
3. The human opens the provisioning frontend and creates the owner passkey.
4. The backend binds the owner artifacts and computes the counterfactual wallet address.
5. The wallet is funded onchain with the minimum activation balance.
6. The backend marks the wallet as ready.
7. The agent can use the wallet through the policy-gated runtime path.

In the current implementation, the externally visible statuses are:

- `created`: wallet request exists, owner not yet bound,
- `owner_bound`: passkey owner is attached, funding or final readiness still pending,
- `ready`: wallet can be used on the runtime path, even if the smart wallet has not yet been deployed onchain,
- `failed`: reserved in the model, but not a major exercised path in the current implementation.

## What Runtime Access Means

Guardrail does not give the agent an unrestricted signer.

The runtime path currently means:

- the wallet is a Kernel smart account,
- the owner path uses a passkey validator,
- the runtime path uses a weighted ECDSA validator,
- the runtime validator is configured as `2-of-2`,
- one signer is the local agent key,
- one signer is the Guardrail backend signer.

So a runtime action only goes through when:

- the agent signs it,
- the backend agrees to sign it,
- the action matches the allowed runtime policy.

## What Policy Guardrail Enforces Today

Guardrail is deny-by-default on the `agent + backend` runtime path.

The current product only supports two policy mechanisms:

- a non-USDC contract allowlist defined by contract address and allowed method selectors,
- a dedicated official USDC policy with an allowed operation list and a sliding budget window.

### Generic Contract Calls

For non-USDC calls, Guardrail currently allows only:

- specific contract addresses,
- specific method selectors on those contracts,
- zero native value.

If the contract is not allowlisted, the selector is not allowlisted, or native value is attached, the backend denies the action.

### Official USDC

Official USDC is treated separately from the generic allowlist.

The current supported official USDC operations are:

- `transfer`
- `approve`
- `increaseAllowance`
- `permit`
- `transferWithAuthorization`

The USDC policy also includes a sliding budget window:

- `daily`: trailing 24 hours,
- `weekly`: trailing 7 days,
- `monthly`: trailing 30 days.

When the backend approves one of those supported USDC actions, the authorized amount is recorded against the budget immediately.

### Typed Data

Typed data are not broadly supported.

In the current product shape, typed-data signing is effectively a narrow runtime surface used for supported official USDC flows, not a general-purpose arbitrary signing feature.

## Supported Product Flows Today

The strongest supported flows in the current codebase are:

- creating a wallet for one agent workflow,
- provisioning the owner through a passkey,
- funding and activating the smart wallet,
- making allowlisted contract calls,
- using official USDC within a configured budget,
- paying x402-protected resources through the exact `eip3009` USDC flow.

The x402 flow is important because it shows the intended wedge clearly: an autonomous agent can pay for protected resources without holding a free-spending wallet key.

## What The Frontend Is And Is Not

The frontend currently has two roles:

- public homepage / product framing,
- provisioning surface for the human owner.

It also publishes a machine-readable `skill.md` endpoint that explains how an agent should set up a Guardrail wallet end to end.

It is not currently the primary runtime control surface for agents.

The CLI is the main runtime entry point for creating wallets, waiting for readiness, making calls, signing supported payloads, and completing x402 flows.

## Supported Chains Today

The current supported chains are:

- Base (`8453`)
- Base Sepolia (`84532`)

Chain support is narrow on purpose because the policy and official USDC handling are chain-aware.

## Trust Model

Guardrail is not claiming to remove trust from the backend.

The current trust model is:

- the human passkey is the durable owner path,
- the local agent key alone is insufficient,
- the backend signer alone is insufficient for the runtime validator,
- the backend operator is trusted to enforce policy correctly,
- the backend operator is also trusted for availability of the runtime service.

This means the main product promise is not "trustless wallet infrastructure."

The main product promise is:

- the agent does not get a freely reusable fully privileged wallet secret,
- the runtime path stays constrained by an explicit backend-enforced policy,
- ownership stays with the human through the passkey path.

## What Is Intentionally Out Of Scope Right Now

The current product is intentionally narrower than a general wallet platform.

Today it is not trying to be:

- a general arbitrary-signing service,
- an unrestricted smart-account execution platform,
- a wallet UI for broad portfolio management,
- a multi-chain policy engine across many networks,
- a fully trust-minimized backendless custody model.

The current shape is much more specific:

- one agent-oriented wallet flow,
- one human owner path,
- one guarded runtime path,
- one narrow policy surface,
- one clear payment wedge through official USDC and x402.
