# x402 Payments

[Docs index](README.md) | [Quickstart](quickstart.md) | [CLI reference](cli.md) | [Runtime policy](runtime-policy.md)

Conduit can be used to pay for protected resources over x402 without giving the agent a fully privileged wallet key.

The current flow is built around exact `eip3009` settlement on official USDC.

## Why This Matters

For agent products, one important use case is letting an agent pay for premium data, APIs, or tools at runtime.

The unsafe version is to give the agent a normal hot key and let it spend freely.

The Conduit version is:

- the human owns the wallet through a passkey
- the agent runs the payment flow
- the backend only co-signs official USDC operations that match the wallet policy

## Main Commands

Build a `PAYMENT-SIGNATURE` header from a ready wallet:

```bash
conduit-wallet x402-sign wal_123 \
  --payment-required-header eyJ4NDAyVmVyc2lvbiI6Mn0=
```

Fetch a resource and automatically complete the challenge when needed:

```bash
conduit-wallet x402-fetch wal_123 \
  https://api.example.com/premium-data
```

## End-To-End Flow

1. The resource server responds with a `402` challenge.
2. The CLI reads the `PAYMENT-REQUIRED` header.
3. Conduit selects a compatible exact `eip3009` requirement for the wallet chain.
4. The CLI prepares the official USDC `TransferWithAuthorization` typed data.
5. The backend checks the wallet policy and co-signs only if the action is allowed.
6. The CLI sends the `PAYMENT-SIGNATURE` header and retries the request.
7. If settlement succeeds, the resource becomes available.

## Policy Interaction

x402 is not a separate trust model. It still uses the same runtime policy layer.

That means:

- official USDC support must be enabled on the wallet
- `transferWithAuthorization` must be allowed in `--usdc-allow`
- the authorized amount still consumes the configured USDC budget

Example wallet creation:

```bash
conduit-wallet create \
  --chain-id 84532 \
  --usdc-period daily \
  --usdc-max 10 \
  --usdc-allow transferWithAuthorization
```

With that policy, the agent can pay protected resources through x402, but only until the wallet consumes `10` USDC in the trailing 24-hour window.

## What Gets Counted

For x402 on the current path, the amount counted against budget is the USDC `value` inside the official `TransferWithAuthorization` payload.

So if an x402 payment authorizes `2` USDC:

- `2` USDC is recorded immediately against the wallet budget
- the next action is denied if it would push the trailing-window total over the configured maximum

For the broader budget semantics, see [Runtime policy](runtime-policy.md).

## Current Scope

The current implementation is intentionally narrow:

- official USDC only
- exact `eip3009` settlement path
- Base Sepolia only

That narrow scope is deliberate. It keeps the runtime policy legible while proving the agent-payment flow end to end.
